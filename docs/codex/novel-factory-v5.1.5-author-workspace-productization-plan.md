## 1. 当前问题诊断

### 1.1 信息架构问题

当前导航是一组**平权菜单**：

```
总览 | 项目 | 创建项目 | 生成章节 | 审核 | 风格 | 配置 | 验收
```

问题：
- **所有页面权重相同**。用户无法区分哪些是主路径、哪些是辅助工具。
- **"生成章节"是独立顶级导航**，暗示它和"项目"是同级概念。但实际上，生成章节是项目内的操作，不应独立存在。
- **"验收"放在面向作者的工作台导航中**，这是开发/测试视角的残留。作者不需要看到 capability matrix。
- **"创建项目"独立导航**，但用户大多数时间不需要创建项目，只需要在已有项目中创作。

### 1.2 主流程割裂问题

当前用户完成一次"生成→查看"需要经过 3 个独立页面：

```
/run → 选择项目、选择章节、点击生成 → 看到结果卡片
→ 点击"查看正文"跳转到 /projects/:id/chapters/:num
→ 点击"查看工作流"跳转到 /runs/:runId
```

每一步都是一次**页面跳转**，丢失上下文：
- 在 ChapterReader 看不到工作流
- 在 RunDetail 看不到正文
- 在 Run 页面生成完就被迫离开

这不是"创作流"，而是"API 测试流"。

### 1.3 项目页表格化问题

`ProjectDetail.tsx`（`/projects/:id`）当前结构：

```
[项目概览卡片: 类型/总章节/总字数/待审核]
[NextAction 卡片]
[章节进度: StatusBadge 统计条 + 章节表格]
[最近运行表格]
```

问题：
- **章节列表是表格**，不是可导航的目录。用户想读某章正文需要在一行里找到小按钮点击。
- **没有正文预览**。项目页应该是"看小说的地方"，但现在只能看到表格行。
- **最近运行占了大量篇幅**，但运行记录是过程数据，不应该和创作内容争夺视觉优先级。

### 1.4 Run 页面孤岛问题

`/run` 是当前生成章节的**唯一入口**：

- 它是一个**独立表单页**：选项目 → 选章节 → 点生成 → 看结果
- 用户在项目工作台点击"生成章节"→ 跳转到 `/run`
- 在 ChapterReader 点击"生成本章"→ 跳转到 `/run`
- 在 Dashboard 点击"生成章节"→ 跳转到 `/run`

问题：
- **所有路径都导向一个独立表单页**，而不是在上下文中完成操作。
- Run 页面应该是一个**高级运行面板**（配置字数、选择 agent、查看详细参数），而不是日常生成入口。
- 日常生成应该是"在项目里点一下就能生成"。

### 1.5 工作流有步骤但缺少产物感

v5.1.4 新增的 `RunDetail.tsx` 展示了 5 个步骤的执行状态：

```
✓ 编剧 - 规划章节场景和情节
✓ 执笔 - 撰写章节正文
✓ 润色 - 优化文字表达
✓ 审核 - 检查内容质量
✓ 发布 - 发布章节内容
```

问题：
- **只有步骤名称和状态**，没有展示每个 Agent 实际产出了什么。
- 用户看到"编剧已完成"，但不知道编剧写了什么剧本大纲。
- 用户看到"执笔已完成"，但不知道草稿 v1 和润色后 v2 的差异。
- 这不是"工作流产物"，而是"状态时间线"。

### 1.6 Review / Style 页价值不清

**Review 页**（`/review`）：
- 当前 stub 模式下，章节直接 `published`，不经过 `review` 状态
- 因此审核队列**永远为空**
- 空状态文案："当前流程下章节生成后会直接发布。已发布章节可在项目工作台查看正文。"
- 这个说明虽然正确，但用户会想："那这个页面存在的意义是什么？"

**Style 页**（`/style`）：
- 需要 style 相关表存在才有数据
- 空状态："暂无风格数据。生成章节或初始化风格后会出现数据。"
- 但用户实际上不知道怎么"初始化风格"
- 这是一个"先有鸡还是先有蛋"的问题

### 1.7 配置中心不够诊断化

当前 `/settings` 包含：
- 运行模式展示
- 启动命令示例
- 配置诊断（LLM 档案/默认 LLM/运行模式）
- LLM 档案表格
- Agent 路由表格
- 配置草案生成器

问题：
- **页面是"设置参考页"**，不是"能不能生成"的诊断页
- 诊断信息散落在不同卡片中
- 缺少一个**顶部的明确答案**："当前能否真实生成？"——Yes/No + 原因
- 配置草案生成器在页面最底部（需要滚动），但它是最有价值的交互

### 1.8 演示模式与真实模式的心理边界

当前状态：
- Layout 顶部有"演示模式"或"真实模式"徽章
- Run 页面有演示模式提示
- ChapterReader 有"演示正文"标签
- RunDetail 有演示模式提示

问题：
- **提示分散在各处**，但用户没有形成一个清晰的心理模型
- 用户在 stub 模式下看到"生成速度快，内容不代表真实质量"的提示，但没有一个地方集中展示"你现在在演示模式，切换需要做什么"
- 缺少一个"模式切换检查清单"

### 1.9 作者控制感不足

当前用户操作完后会问：
- "这一章质量怎么样？" —— 不知道，没有集中展示质量维度
- "下一步该做什么？" —— 需要自己判断，NextAction 只给一个建议
- "有没有需要我处理的问题？" —— 需要分别去 Review、Style、Settings 检查
- "现在生成的是真实内容还是演示内容？" —— 需要看顶部小徽章

**作者控制感 = 一眼可知当前状态 + 一步可达下一步 + 一键可干预**

当前系统三个都欠缺。

## 2. 新产品定位

### 这不是什么

| 不是 | 原因 |
|------|------|
| 后台管理系统 | 不应该以 CRUD 表格为主要交互模式 |
| API demo / 测试面板 | 不应该暴露内部状态枚举和 capability ID |
| CLI 的 Web 包装 | 不应该只是把 CLI 参数变成 HTML form |
| 验收控制台 | 验收矩阵对作者无意义，应从主路径中移除 |

### 这是什么

> **以项目为中心、以章节为主线、以工作流为辅助的作者创作工作台。**

具体来说：
- **小说生产驾驶舱**: 进入系统就知道当前小说状态、下一章是什么、有没有问题需要处理
- **写作 IDE**: 章节是主要工作对象，可以阅读、编辑、生成、返修
- **Agent 协作中心**: 能看到每个 AI Agent 产出了什么，而不是只知道它们"跑完了"

### 核心产品心智

用户打开系统后，第一眼应该能回答：
1. 当前是哪本小说？ → 顶部/侧栏项目上下文
2. 已写到第几章？ → 章节导航
3. 下一章该做什么？ → 下一步建议
4. 最近生成了什么？ → 最近活动
5. 是否有审核/返修/配置问题？ → 状态诊断
6. 当前是演示还是真实模式？ → 全局模式指示器
7. 生成内容是 Stub 还是 LLM？ → 内容来源标签

## 3. 新信息架构

### 3.1 页面结构重设计

```
主路径（高频、核心）
├── /                        创作中心（代替 Dashboard）
├── /projects/:id            项目工作台（三栏布局）
│   ├── (内嵌)               章节阅读/工作区
│   ├── (内嵌)               快速生成
│   └── (内嵌)               工作流查看
│
辅助路径（中频、工具）
├── /projects                项目列表（卡片化）
├── /projects/:id/run        高级运行面板（代替独立 /run）
├── /review                  审核工作台
├── /style                   风格管理

│
弱化路径（低频、开发用）
├── /onboarding              创建项目（流程中）
└── /acceptance              验收矩阵（开发/测试用，从主导航移除）

## 4. 项目工作台三栏设计

这是 v5.1.5 最核心的设计变更。

### 4.1 三栏布局

```
┌──────────────────────────────────────────────────────────────┐
│ ← 返回项目列表    [演示模式]                    [项目设置]   │
│                                                              │
│ 斗破苍穹 · 第 3 章 / 共 10 章    [生成当前章] [高级运行]   │
├──────────┬───────────────────────────────────┬───────────────┤
│          │                                   │               │
│ 章节导航 │        章节主内容区                │  上下文侧栏   │
│          │                                   │               │
│ 1.陨落   │  [正文] [工作流] [产物] [历史]    │  下一步建议   │
│   ✓ 已发布│                                  │               │
│          │  "第三章 遭遇强敌"                │  当前章: 3    │
│ 2.初试   │                                   │  状态: 已发布 │
│   ✓ 已发布│  正文内容...                     │  字数: 3,200  │
│          │  正文内容...                      │               │
│ 3.遭遇   │  正文内容...                      │  最近运行     │
│   ● 当前 │  ...                              │  ✅ completed │
│          │                                   │  查看工作流   │
│ 4.暗流   │                                   │               │
│   ○ 已规划│                                  │  继续生成     │
│          │                                   │  [生成下一章] │
│ 5.(空)   │                                   │               │
│   - 未规划│                                  │  质量状态     │
│          │                                   │  质量分: 85   │
│          │                                   │  风格: 已对齐 │
│          │                                   │               │
│          │                                   │  演示模式     │
│          │                                   │  内容由本地   │
│          │                                   │  Stub 生成    │
│          │                                   │               │
├──────────┴───────────────────────────────────┴───────────────┤
│ [上一章] [下一章] [重新生成] [导出]                          │
└──────────────────────────────────────────────────────────────┘
```

### 4.2 左侧栏：章节导航

**设计原则**：
- 显示所有章节的紧凑列表
- 每行：章节号 + 状态图标 + 标题（如果有）
- 当前选中章高亮
- 点击切换章节
- 状态用颜色和图标区分，不依赖文字

**状态视觉语言**：
| 图标 | 含义 |
|------|------|
| ✓ (绿色) | 已发布 |
| ✎ (蓝色) | 有内容但未发布 |
| ○ (灰色) | 已规划，未生成 |
| ⚠ (黄色) | 阻塞/需审核 |
| ✗ (红色) | 生成失败 |
| - (浅灰) | 未规划 |

**交互**：
- 点击章节 → 中间栏切换到该章节
- 当前章节高亮，有明显视觉指示
- 滚动列表，当前章节始终可见（scroll into view）

### 4.3 中间栏：章节主内容区

**首屏默认状态**（无选中章节时）：
- 显示项目简介
- 显示"开始创作"引导
- 显示最近活动摘要

**Tab 设计**：

| Tab | 标签 | 何时可见 | 内容 |
|-----|------|---------|------|
| 正文 | 正文 | 始终 | 未生成时显示生成入口；已生成时显示阅读视图 |
| 工作流 | 工作流 | 有运行记录时 | 当前 v5.1.4 RunDetail 的步骤时间线 |
| 产物 | Agent 产物 | 有产物数据时 | 编剧大纲/执笔摘要/润色对比/审核报告 |
| 历史 | 版本历史 | 有多个版本时 | 版本列表，可对比 |

**正文 Tab（默认）**：

未生成状态：
```
┌─────────────────────────────────┐
│                                 │
│        第 4 章                  │
│        暗流涌动                 │
│                                 │
│    本章尚未生成                 │
│    编剧将规划场景和情节         │
│    执笔将撰写章节正文           │
│                                 │
│    [生成本章]                   │
│                                 │
│    预计字数: 2,000-4,000        │
│    生成模式: 演示模式           │
│                                 │
└─────────────────────────────────┘
```

已生成状态：
```
┌─────────────────────────────────┐
│ 第 3 章 遭遇强敌                │
│                                 │
│ 来源: 演示 | 字数: 3,200        │
│ 生成时间: 2026-04-27 10:00      │
│ ─────────────────────────────── │
│                                 │
│ 正文内容（当前 ChapterReader    │
│ 的阅读视图，720px 最大宽度，    │
│ 1.9 行高）                      │
│                                 │
└─────────────────────────────────┘
```

### 4.4 右侧栏：上下文侧栏

**固定元素**（始终可见）：
1. **下一步建议** — 动态内容，根据当前状态推荐
2. **当前章节状态** — 章节号、状态、字数、来源
3. **生成操作** — 生成本章/生成下一章按钮
4. **模式提示** — 演示模式/真实模式 + 简短说明

**条件元素**（有数据时显示）：
5. **最近运行** — 最后一次运行的状态 + 查看工作流链接
6. **质量状态** — 质量分 + 风格对齐状态
7. **需要关注** — 阻塞/审核提醒

**侧栏优先级**（从上到下）：
1. 模式提示（最重要，不可折叠）
2. 下一步建议（次重要）
3. 生成操作
4. 当前状态
5. 最近运行
6. 质量/风格

### 4.5 响应式策略

- **>1280px**: 三栏完整显示
- **1024-1280px**: 右侧栏可折叠，默认展开
- **<1024px**: 左侧章节导航折叠为汉堡菜单，中间栏占满宽度，右侧栏折叠
- **<768px**: 单栏模式，章节导航弹出 overlay

## 5. 生成交互重构

### 5.1 为什么 `/run` 不应再是主入口

当前 `/run` 的问题：
- 它是一个**上下文脱离**的独立页面：你需要离开项目工作台才能生成
- 它是一个**表单思维**的产物：选择 → 配置 → 提交 → 看结果
- 它暗示"生成章节"是一个需要专门去做的独立任务，而不是创作流程的自然一环

### 5.2 新生成体验设计

**日常生成（项目工作台内）**：

在项目工作台右侧栏中：
```
[生成本章]  ← 主要操作，一键生成当前章节
[生成下一章] ← 快捷操作
```

点击后：
- 不跳转页面
- 中间栏切换到"工作流" Tab，显示实时步骤
- 右侧栏更新为生成进度
- 生成完成后自动切换到"正文" Tab 显示结果

**快速生成弹窗**（可选配置）：

```
┌──────────────────────────────────┐
│ 生成第 4 章 - 暗流涌动           │
│                                  │
│ 已选择默认配置：                 │
│ • 章节: 4                        │
│ • 字数目标: 2,000-4,000          │
│ • 生成模式: 演示模式             │
│ • 简短指令: (可选)               │
│  [________________________]      │
│                                  │
│     [取消]      [开始生成]       │
└──────────────────────────────────┘
```

**高级运行面板**（`/projects/:id/run`）：

保留高级运行页面，用于：
- 自定义字数目标
- 选择 Agent 配置
- 添加写作指令/风格覆盖
- 批量运行多章

## 6. 工作流产物感设计

### 6.1 当前状态 vs 目标状态

**当前**（v5.1.4）：
```
✓ 编剧 - 规划章节场景和情节
✓ 执笔 - 撰写章节正文
✓ 润色 - 优化文字表达
✓ 审核 - 检查内容质量
✓ 发布 - 发布章节内容
```

**目标**：
```
✓ 编剧 - 规划章节场景和情节
  └ 产出: 场景节奏表 / 冲突设计 / 钩子设置
  
✓ 执笔 - 撰写章节正文
  └ 产出: 草稿 (3,200 字) / 关键段落 / 对话摘要
  
✓ 润色 - 优化文字表达
  └ 产出: 修改摘要 (12 处修改) / 风格调整说明
  
✓ 审核 - 检查内容质量
  └ 产出: 质量报告 / 评分 85 / 问题 0 / 通过 ✓
  
✓ 发布 - 发布章节内容
  └ 产出: 最终正文 / 版本 v1 / 发布时间 2026-04-27
```

### 6.2 各步骤产物定义

## 7. 章节详情重构

### 7.1 当前 ChapterReader 分析

当前 `ChapterReader.tsx` (`/projects/:projectId/chapters/:chapterNumber`)：
- 独立页面，脱离项目上下文
- 只有正文阅读功能
- 导航需要加载 workspace
- "查看工作流"是外部链接

### 7.2 新设计：嵌入还是独立？

**决策：项目工作台的展开模式**

`ChapterReader` 不再是一个独立路由页面，而是**项目工作台中间栏的一种视图状态**。

- URL 方案：`/projects/:id?chapter=3` （查询参数，不改变路由层级）
- 或者保留旧路由作为快捷入口，但重定向到项目工作台

**推荐方案**：保留 `/projects/:projectId/chapters/:chapterNumber` 作为直接入口，但页面组件共享项目工作台的中间栏逻辑，让用户在需要时可以展开为全屏。

### 7.3 三种展示模式

| 模式 | 触发条件 | 效果 |
|------|---------|------|
| 嵌入模式 | 在 `/projects/:id` 中点击章节 | 中间栏切换内容，左侧栏保持可见 |
| 展开模式 | 直接访问 `/projects/:id/chapters/:num` | 全屏阅读体验，左侧栏可折叠 |
| 专注模式 | 用户点击"全屏阅读" | 纯阅读界面，所有 chrome 隐藏 |

### 7.4 嵌入模式（默认）

在项目工作台中，中间栏 = ChapterReader + 工作流 + 产物 + 历史。

不需要单独跳转。

### 7.5 展开模式（独立 URL）

当用户通过外部链接或书签直接访问 `/projects/:id/chapters/:num` 时：
- 显示全宽布局
- 左侧保留紧凑的项目章节导航（可折叠）
- 主要内容区包含 Tab 切换
- 顶部有"返回项目工作台"链接

     ✓ 执笔 - 已完成
     ✗ 润色 - 失败
       错误原因: Stub simulation error
   → 右侧栏：
     下一步建议: "重新生成第 4 章"

3. 点击"重新生成"
   → 重新运行章节
   → 成功
```

### 11.5 路径五：从演示模式切换到真实模式

```
1. 在项目工作台中注意右侧栏：
   "演示模式 - 内容由本地 Stub 生成"

2. 点击"了解详情"或顶部导航 → 配置中心

3. /settings
   → 顶部诊断卡：
     "❌ 无法真实生成"
     "原因: 演示模式 + 未配置 LLM"

4. 按照诊断卡指示：
   a. 使用配置草案生成器生成 YAML
   b. 复制到 config/local.yaml
   c. 设置环境变量

5. 切回终端，重启服务：
   $ export OPENAI_API_KEY=sk-xxx
   $ novelos api --config config/local.yaml --llm-mode real

6. 刷新浏览器
   → 创作中心顶部变为"真实模式"
   → 项目工作台右侧栏变为"真实模式"
   → 生成时调用真实 API
```
